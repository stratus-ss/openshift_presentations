OpenShift 101: Kubernetes Fundamentals
From A Red Hat Perspective

 



Steve Ovens
Sr. SRE Architect
June 2025

* we used to deploy monoliths on physical machines

* we moved deployment monoliths in vms

* we solved the scaling problem by deployment multiple VMs

* we then moved to breaking up the application

* About the same time microservices were born

At the end of the day, no one actually cares about the platform

The focus on SRE is all about the end user experience

Traditional Operations KPIs (mostly) no longer apply

reorients the focal point away from individual systems and performance towards the goal of making end user happy

* combine microservices with Linux namespaces and we moved into the container era

* Docker came along and it was great!

* but ultimately it did not scale or self heal the way that the industry wanted

* how do you manage thousands of containers at scale across multiple servers?

so what are the next steps?

So now that you've picked a general direction you have a decision to make...

What kubernetes distribution do you install?

* Kubernetes can be very flexible

* It's exciting and interesting to assemble all the pieces

* Especially if you are following a use case similar to your own

* What about at repeatable scale or if you can't find a similar use case?

* * Which Networking stack do I use? How do I handle ingress? What do I do for logging? Monitoring? Alerting?

* OpenShift has a few installation methods

* IPI for the easy mode

* UPI for more customization

* Red Hat tests all the components and ships them as a release

* There is a support team and thousands of businesses running your configuration

* Kubernetes requires you to manage the upgrades of all the components

* So what do you have after the cluster is installed?

* Generally speaking you will have 3 control plane nodes

* * These have scheduler, ETCD, API, Controller Manager etc

* You will have N number of worker nodes that host the actual applications

* Runs on every worker node.

* Responsible for joining the node to the cluster.

* Establishes a connection with the control plane.

* Runs containers on the node as specified by pod specs.

* Ensures containers are running and healthy (using health checks like liveness and readiness probes)

* In a restaurant there can be many chefs each one responsible for a specific section of the kitchen

* * The kubelet is the prep cook. They know what food is being ordered and their job is to ensure that the raw ingredients start to be processed

* Node tries to join, issues certificate signing request

* An admin needs to approve this to secure the cluster

* Side note: certs are used for the internal cluster communication

* In a restaurant, this is essentially like requiring that all of the staff have badges with photo IDs

* Everything is an API call

* From the UI to the OC command line utility

* Ultimately no one really cares about installing and managing a cluster

* We want to run applications!

* smallest unit in k8s

* abstraction around containers so that you can talk in a standard way, regardless of the container or container run time

* usually only 1 application per pod

* Pods share resources such as volumes and IP addresses

* Complex applications may consist of multiple pods

* Pods are ephemeral by nature

* In a restaurant, pods are the chefs who combine all the wonderful ingredients to make you a meal!

* * The do all the work of gathering the raw foods, preparing, cooking and plating the food

* Declares desired state for pods

* Includes things like:

* * Networking

* * Storage

* * Number of replicas

* * Health checks

* * Number of replicas

* * Resource Limits

* In a restaurant, a deployment is like a note to the chef about how the customer wants a certain dish.

* * If there are multiple people at the same table who want the dish, it also includes the number of servings to make

* Looks at cpu/ram/affinity and places the pod on a node

* * Filters for appropriate nodes

* * Ranks the nodes

* * Assigns node to highest ranking node

* In a restaurant the scheduler is the person who takes reservations and decides which party will be seated at which table

Cluster Networking

Kubernetes Ingress

* Used to get traffic into the cluster.

* You need to setup your own controller (NGINX, Traefik etc)

* The features you get are based on your ingress controller

* Requires you to manage certificates (likely through something like CertManager)

OpenShift Routes

* Uses HA-Proxy.

* Tight integration and just works!

* Can support TLS Edge, Passthrough of re-encrypt

* Nice integration with the web UI for quick adjustments

* A route is like a menu that you pick your items from

* * When you order a specific dish it may have many things that need to be done before the end product is available

* * Each dish might have special chefs assigned to making them

* while every pod gets an ip, that ip is not constant

* therefore a service is like an internal dns entry

* except multiple pods can be part of the same service

* Its a proxy that runs on every node

* implements kubernetes service

* In a restaurant, this is like the computer program that takes the orders from the wait staff (services)

* * this computer program ensures it is displayed on the correct screen in front of the appropriate chef

* pods are ephemeral, that means if they get deleted the data disappears

* default storage is emptyDir unless otherwise specified

* you need to define Persistent Volumes

* * This essentially defines where to find the storage

* these can be fc, hostPath, iscsi, local, nfs or csi (more on this later)

* You can define capacity metadata like size of disk

* You can indicate filesystem or block

* You can choose from:

* * ReadWriteOne (RWO): Can only be mounted to a single pod at a time

* * ReadOnlyMany (ROX): ReadOnly to multiple pods at the same time

* * ReadWriteMany (RWX): Multiple pods can read and write (side note, this is required for VM migration

* IMPORTANT: the actual mode of the storage is enforced by the storage backend not Kubernetes!

* Claims make a request of the storage system

* The system matches the metadata provided by either PVs or the CSI provider

* Claims are then referenced in the deployment specification when deploying an application

* This will be familiar to anyone who has worked with volumes in Docker

* CSI drivers let kubernetes know how to interact with software defined storage

* S3 (minio), OpenShift Data Foundation, NetApp and so on are examples

* The system requests storage from the CSI driver which creates volumes on demand

* This is done by creating a PVC that has the storage class defined in it

* There has to be an easier way to update a container's contents...

* You may want to make a container more generic

* You can do this by injecting the configuration into the container

* This is essentially mounting a file overtop of an existing one (or into a directory)

* You don't have to rebuild a container to update a configuration file!

* You can have an entire config file, env variables or even binary data

* in a restaurant if you have instructions to cook eggs

* * You might have the time each side should be cooked written down. By increasing or lessening the time you get over-easy, over-med or over-hard

* Sometimes you need to inject sensitive information into your pod

* Sometimes you need white-space accurate data in your pod

* Secrets do not reside on the node's disk (tmpfs)

* They are very similar in use to configMaps

* The dirty secret about secrets is, they are not secret

* * they are just base64 encoded

* In a restaurant, secrets are like recipes for special dishes that are locked up in the office

* * Anyone who has a key to the office can read the recipes

* You probably should use a centralized service to manage secrets

* Hashicorp Vault is the standard for secret management

* There is tight integration with OpenShift

* There is a version up on GitHub. It's a source available license

* * However 4 years after release it changes to the Mozilla Public License 2.0

* ESO vs VSO

* VSO can rotate secrets and redeploy

* VSO can revoke tokens/credentials

* VSO has its own prometheus metrics

* Has multiple types of secrets (KV, dynamic, pki)

* SELinux enforces access controls on all processes within OpenShift only giving access resources they are explicitly allowed to use

* Even if a container is compromised, SELinux prevents it from accessing sensitive files or processes outside its designated boundaries

* SELinux implements the principle of least privilege, automatically restricting applications and processes to only what they need to function.

* By default, OpenShift labels and isolates all container processes and files

* You can indicate filesystem or block

* You can choose from:

* * ReadWriteOne (RWO): Can only be mounted to a single pod at a time

* * ReadOnlyMany (ROX): ReadOnly to multiple pods at the same time

* * ReadWriteMany (RWX): Multiple pods can read and write (side note, this is required for VM migration

* IMPORTANT: the actual mode of the storage is enforced by the storage backend not Kubernetes!

* In a restaurant, imagine that each chef has their own work station with their own ingrediants.

* * SELinux is there to make sure that a chef cannot go take things from another work station

* K8s: Root Allowed By default

* K8s: can bind to any port if root container

* K8s: non-restrictive by default

* OCP: non-root, random UID above 1000000 by default

* OCP: More secure by default, need to take specific action to run as non-root

* Think of a restaurant

* Requests = Your Reservation

* * You book a table for a certain number of people

* * The kube scheduler decides which table in the restaurant has enough seats for your party

* Limits = Maximum party size

* You cannot have more people at your table than you have in your reservation

* * If one table attempts to seat too many people, they are asked to leave (the pod is killed)

* * This prevents one party from overwhelming the entire restaurant

* ETCD is a clustered database that tracks everything about the cluster

* Every query in the cluster goes to ETCD as the source of truth

* in a restaurant, ETCD is like the ledger of all of the current details of patrons in the building

* * what food has been delivered

* * how many tables you have (and how many are full)

* This is a broad term used to describe many discrete controllers (replication, node, service etc)

* You can think of this like a restaurant manager

* The wait staff (the API) input the orders into the computer (etcd)

* * The manager looks at the orders on the computer

* * But they also check with the cooks to see what has actually been prepared

* * If an order needs to be filled, or if a plate is spilled, the manager assigns the task of making the food to a chef (the scheduler)

* * This way the staff stay efficient and only have to worry about fulfilling the direct duties of their job
